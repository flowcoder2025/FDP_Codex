# Layer 2 Scope Code Options Packet

Status: options-packet.

WI: WI-CX0034-docs.

Decision target: Layer 2 project scope code rule.

## Boundary

This packet does not choose the Layer 2 project scope code rule. It prepares options for user selection before the first Layer 2 target-project scaffold.

No target-project scaffold is generated by this packet. No public API or external contract is stabilized by this packet.

## Problem

Layer 2 target projects need WI identifiers that remain separate from FDP_Codex Layer 1 identifiers.

Layer 1 uses:

```text
WI-CXNNNN-category
```

Layer 2 must not reuse `CX`. The scope code must let Codex, users, and future contributors quickly distinguish target-project work from FDP_Codex repository work while keeping names short enough for branches, PRs, issues, and handoffs.

## Constraints

A valid rule should satisfy these constraints:

- Preserve Layer 1 and Layer 2 separation.
- Avoid encoding severity, risk, or category into the scope code.
- Keep WI ids sortable and compatible with existing `WI-<scope><number>-<category>` shape.
- Work for small scratch projects and larger OSS/product repositories.
- Avoid leaking private target-project history into Layer 1 records.
- Require registration in the target manifest and Layer 1 provenance record before target work starts.
- Leave collision handling explicit rather than implicit.

## Option A: User-Chosen Mnemonic Code

Rule:

- At scaffold creation, the user chooses a 2-6 character uppercase alphanumeric project code.
- The code must start with a letter.
- `CX` is reserved for FDP_Codex Layer 1 and cannot be used by Layer 2.
- The target manifest records the chosen code and the reason.

Example:

```text
WI-FS0001-docs
WI-FDPAPP0001-fix
```

Tradeoffs:

- Strong human readability.
- Lowest surprise for OSS contributors because the code can match the repo, product, or organization language.
- Requires a user decision before first scaffold generation.
- Needs a collision check when one FDP_Codex workspace manages multiple target projects.

Recommended use:

- Named repositories, public projects, long-running apps, and projects likely to receive contributors.

## Option B: Temporary Generic Code

Rule:

- Use `TG` as the temporary target-project scope code until the user chooses a project-specific code.
- The target manifest must mark the code as temporary.
- A later migration must rename outstanding target WIs before public release or external contribution mode.

Example:

```text
WI-TG0001-docs
WI-TG0002-fix
```

Tradeoffs:

- Fastest for prototypes and very small projects.
- Keeps Layer 1 and Layer 2 separated without waiting on naming.
- Becomes ambiguous if multiple target projects exist in the same workspace.
- Creates migration debt if the project becomes public or long lived.

Recommended use:

- Scratch work, demos, private experiments, and cases where the user has not named the target project yet.

## Option C: Repository-Derived Code

Rule:

- FDP_Codex derives the code from the target repository or package name.
- The derived code must still be shown to the user or recorded in a decision packet before scaffold generation.
- Collision handling must be explicit.

Example:

```text
flow-set -> FS -> WI-FS0001-docs
fdp-app -> FA -> WI-FA0001-docs
```

Tradeoffs:

- Minimizes user input for common repositories.
- Good default when repository naming is stable.
- Can produce weak or confusing codes for generic names such as `app`, `client`, `server`, or `web`.
- Renames can make the code feel stale.

Recommended use:

- Repositories with stable, distinctive names where the generated code is obvious.

## Option D: Long Explicit Project Slug

Rule:

- Use a longer uppercase slug, 4-10 characters, when a short code would be ambiguous.
- Keep the same WI shape with the slug before the number.

Example:

```text
WI-FLOWSET0001-docs
WI-FDPAPP0001-docs
```

Tradeoffs:

- Very low ambiguity.
- Best for portfolios with many similar projects.
- Longer branch, PR, and issue names.
- Less ergonomic for repeated manual references.

Recommended use:

- Multi-project portfolios, generated target projects with similar names, or cases where a short code would collide.

## Recommended Default

Recommended primary choice: Option A.

Option A is the best default for serious target projects because it is explicit, readable, and stable. It respects the user gate and avoids Codex silently choosing a public naming convention.

Recommended fallback: Option B.

Option B is acceptable only when speed matters more than long-term traceability. It should carry migration debt and a repayment trigger before public release, external contribution mode, or long-lived automation.

## Proposed Acceptance Rule

If the user chooses Option A, the first Layer 2 scaffold may use this rule:

```text
WI-<PROJECT_CODE><NNNN>-<category>
```

Where:

- `PROJECT_CODE` is 2-6 uppercase alphanumeric characters.
- `PROJECT_CODE` starts with a letter.
- `PROJECT_CODE` is not `CX`.
- `NNNN` is a zero-padded sequence number scoped to the target project.
- `category` uses the FDP-compatible category set unless the target manifest explicitly narrows it.
- The target manifest records `project_scope_code`, `scope_code_status`, and `scope_code_decision_ref`.
- Layer 1 provenance records the FDP_Codex WI, commit, and manifest chunks that generated the scaffold.

## Decision Needed

The user must choose one of these paths before first Layer 2 scaffold generation:

- A: user-chosen mnemonic code.
- B: temporary `TG` code with migration debt.
- C: repository-derived code with explicit confirmation.
- D: long explicit project slug.

Until the user chooses, first Layer 2 target-project scaffold generation remains blocked.
